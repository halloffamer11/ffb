🐛 Widget Layout Persistence Debug

Comprehensive debugging for the widget layout persistence issue

❌ Issue Identified

Problem: Widget layouts are not persisting across page refreshes.

Symptom: When widgets are moved and the page is refreshed, they return to default positions.

Expected: Widget positions should be saved and restored across sessions.

🔍 Debug Testing Steps

1

Open React App & Developer Tools

Open the React application and browser developer tools console

🚀 Open React App

Action: Press F12 to open developer tools, click Console tab

2

Check Initial Load Messages

Look for WidgetGrid debug messages on initial load

Expected console messages: 🔧 WidgetGrid: Loading saved layouts... 🔧 WidgetGrid: Saved layouts from storage: null (or actual data) 🔧 WidgetGrid: No saved layouts found, using defaults (or "Merged layouts:") 🔧 WidgetGrid: Test save after load: { ok: true }
3

Test Manual Save Function

Click the "Manual Save" button in the React app

Expected console messages: 🔧 Manual save test: { lg: [...], md: [...], sm: [...] } 🔧 Manual save result: { ok: true }

If this fails: Storage adapter has an issue

4

Enter Edit Mode & Move Widget

Click "Edit Layout" button, then drag a widget (e.g., Player Search) to a new position

Expected console messages: 🔧 WidgetGrid: Layout changed { _layout: [...], allLayouts: {...} } 🔧 WidgetGrid: Complete layouts to save: {...} 🔧 WidgetGrid: Save result: { ok: true } ✅ WidgetGrid: Layouts saved successfully

If no messages: onLayoutChange is not being triggered

5

Check localStorage Directly

Run this command in browser console to check if data was saved

localStorage.getItem('widget-layouts::layouts')

Expected: JSON string with layout data

If null: Data is not being saved to localStorage

6

Test Page Refresh

Refresh the page (F5) and check if widget stayed in new position

Expected console messages after refresh: 🔧 WidgetGrid: Loading saved layouts... 🔧 WidgetGrid: Saved layouts from storage: { lg: [...], md: [...], sm: [...] } 🔧 WidgetGrid: Merged layouts: {...}

Result: Widget should be in the moved position

7

Advanced Debugging

If persistence still fails, run these commands in console:

// Check all localStorage keys Object.keys(localStorage).filter(key => key.includes('widget')) // Test storage adapter directly import('./src/adapters/storage.js').then(({ createStorageAdapter }) => { const adapter = createStorageAdapter({ namespace: 'widget-layouts', version: '1.0.0' }); console.log('Available:', adapter.isAvailable()); console.log('Current layouts:', adapter.get('layouts')); });

🔧 Potential Issues to Check

  • Browser Storage: Private/incognito mode disables localStorage
  • Storage Quota: localStorage might be full
  • React Grid Layout: onLayoutChange might only trigger during drag, not on mount
  • State Management: React state might not be syncing with storage calls
  • Namespace Collision: Multiple storage adapters might conflict

📋 Debug Checklist

🎯 Root Cause Analysis

A

If Manual Save works but drag doesn't save:

Issue: React Grid Layout onLayoutChange not triggering

Solution: Check draggableHandle, isDraggable settings, or React Grid Layout configuration

B

If saving works but loading doesn't:

Issue: Data format mismatch or loading logic problem

Solution: Check data serialization/deserialization and merge logic

C

If neither saving nor loading works:

Issue: Storage adapter or localStorage access problem

Solution: Check browser settings, storage quotas, or adapter configuration

🎯 Expected Outcome

After completing these debug steps, we should identify exactly where the persistence is failing.

Goal: Widget layouts should persist across page refreshes with proper console logging showing the save/load process.
